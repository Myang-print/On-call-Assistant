可以。V3 证据链要修的核心是：**Agent 最终回答里的每一个 source，必须来自 runtime 中一次成功的 `readFile(sop-xxx.html)` 工具调用，而不是绕过工具层直接从 `DocumentStore` 拿内容。**

现在老师指出的问题本质是：

```text
trace:
  readFile(manifest.json)

answer/sources:
  sop-001 / sop-002 / sop-005 ...

但 trace 里没有：
  readFile(sop-001.html)
  readFile(sop-002.html)
  readFile(sop-005.html)
```

这会让 V3 看起来像“表面有工具，实际证据来自旁路检索”。Agent 项目里这是比较严重的问题。

**目标证据链**
修完后，V3 应该变成：

```text
用户问题
  ↓
readFile("manifest.json")
  ↓
根据 manifest + query 选择候选 SOP 文件
  ↓
readFile("sop-001.html") / readFile("sop-002.html") / ...
  ↓
只从成功读取的 SOP 内容生成 retrieved_docs
  ↓
sources 只来自这些成功读取的 SOP
  ↓
AnswerComposer 只基于 retrieved_docs 生成回答
  ↓
trace / sources / answer 三者一致
```

验收不变量：

```python
for source in response["sources"]:
    assert trace_has_successful_read_file(source["filename"])
```

**具体改法**
我建议分 5 层做。

**1. 保留 manifest 读取**
`manifest.json` 仍然是 Agent 的入口，这是合理的。

因为 Agent 不应该一开始就枚举/读取所有 SOP。正确流程是：

```text
readFile("manifest.json")
```

manifest 里每条包含：

```json
{
  "doc_id": "sop-001",
  "filename": "sop-001.html",
  "title": "...",
  "keywords": [...]
}
```

Agent 先用 manifest 做轻量路由。

**2. 新增 SOP 选择器**
在 runtime 或 agent 层做一个 deterministic selector。

输入：

```python
query: str
manifest_items: list[dict]
```

输出：

```python
["sop-001.html", "sop-004.html"]
```

选择规则可以先确定性实现，不必上 LLM：

- query 命中 keywords
- query 命中 title
- query 中英文 token 命中 cleaned keyword
- 可以复用当前 v2/hybrid 的排序思想，但注意：这里只能用 manifest 里的轻量字段，不能直接读取 SOP 正文

例如：

```text
"服务 OOM 了怎么办？"
-> manifest keywords/title 命中 OOM、后端、服务
-> select sop-001.html

"数据库主从延迟超过30秒怎么处理？"
-> 命中 数据库、主从、延迟
-> select sop-002.html
```

这个 selector 可以放在：

```text
app/agent_runtime.py
```

或者更干净一点新建：

```text
app/evidence_selector.py
```

但如果后续严格 SCOPE，需要你授权新文件。

**3. runtime 必须真实调用 readFile(sop-xxx.html)**
这是关键。

不能在 `agent.py` 里直接：

```python
DocumentStore.from_data_dir()
```

然后拿正文。

而应该让 runtime 执行：

```python
readFile("sop-001.html")
```

并把结果写进 trace：

```python
{
  "step": 1,
  "action": {
    "type": "readFile",
    "fname": "sop-001.html"
  },
  "observation": {
    "ok": true,
    "tool": "readFile",
    "bytes": 1234
  }
}
```

同时 runtime 内部构造：

```python
retrieved_docs = [
  {
    "filename": "sop-001.html",
    "title": "...",
    "content": "...readFile返回内容..."
  }
]
```

这一步必须注意：

- `retrieved_docs` 只能来自成功 `readFile`
- `sources` 只能来自这些 `retrieved_docs`
- 如果 `readFile` 失败，不允许加入 source

**4. agent.py 删除/降级旁路 evidence**
现在最大问题大概率在 `agent.py`：

```python
if runtime_docs:
    return runtime_docs, runtime_sources

# 否则从 DocumentStore + hybrid_search 旁路补 evidence
```

这个 fallback 对 v3 证据链是不合规的。

补救版建议改成：

```python
runtime_docs = _extract_retrieved_docs(runtime_result)
runtime_sources = _extract_sources(runtime_result)

if runtime_docs:
    return runtime_docs, runtime_sources

return [], []
```

也就是：**V3 不再绕过工具层补 evidence。**

如果担心用户体验，可以让 API 层 fallback 到 v2，但要明确 trace：

```text
agent evidence insufficient -> rollback v2
```

而不是让 V3 假装自己读过 SOP。

**5. AnswerComposer 只吃 retrieved_docs**
AnswerComposer 本身不要调用工具，也不要查 DocumentStore。

它只接收：

```python
retrieved_docs
sources
trace
```

如果：

```python
retrieved_docs == []
```

返回：

```text
当前没有足够的 SOP 文档支持判断...
```

如果有 docs，则基于 docs 内容输出。

**需要新增的核心测试**
这是老师最关心的部分。

**我建议的最小补救实现**
不要大改 planner，不要接复杂 LLM。

最小可交付方案：

1. `run_deterministic_agent(query)` 支持接收 query。
2. runtime step 0：`readFile("manifest.json")`
3. runtime step 1：根据 manifest 选择 SOP filename。
4. runtime step 2..N：逐个 `readFile(sop-xxx.html)`
5. runtime 返回：
   - `retrieved_docs`
   - `sources`
   - `trace`
6. `agent.py` 不再从 `DocumentStore` 旁路拿 SOP。
7. `AnswerComposer` 用 `retrieved_docs` 生成回答。

这样老师看到的是：

```json
trace: [
  {"action": {"type": "readFile", "fname": "manifest.json"}, "observation": {"ok": true}},
  {"action": {"type": "readFile", "fname": "sop-001.html"}, "observation": {"ok": true}}
],
sources: [
  {"filename": "sop-001.html", "title": "后端服务 On-Call SOP"}
],
answer: "根据 sop-001，建议先检查..."
```

这就是闭环。

**需要小心的点**
不要把 v3 修成“搜索接口换皮”。

也就是说，V3 可以用 manifest 选择候选，但最终证据必须来自：

```text
readFile(sop-xxx.html)
```

不能来自：

```text
DocumentStore.get()
hybrid_search_documents()
search_documents()
```

这些可以用于 v1/v2，但 V3 Agent 的 evidence 不应该绕过工具。

**补救版交付时可以这样描述**
你可以在补救说明里写：

```text
原版本问题：
V3 answer 的 sources 可能来自检索层旁路，而不是 runtime 中成功 readFile 的文件。

修正后不变量：
V3 AnswerComposer 只接收 successful readFile 产生的 retrieved_docs。
sources 中每个 filename 必须能在 runtime trace 中找到对应成功 readFile 调用。
如果没有成功读取 SOP，V3 返回证据不足或 rollback，不生成肯定回答。

新增测试：
- source backed by readFile trace
- OOM -> sop-001.html
- 主从延迟 -> sop-002.html
- 入侵 -> sop-005.html
- 推荐质量下降 -> sop-008.html
- unknown query -> insufficient evidence
```

这会非常直接地回应老师反馈。
